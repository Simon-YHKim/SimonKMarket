# nocode-monetization

코드도, 레포도, 백엔드도 없는 사용자가 **운영 플랫폼에 가입하는 것만으로** 수익을 내도록 안내하는 skill.

대상: 취미·전문성을 가진 은퇴자, 동네 자영업, 1인 크리에이터. "결제 시스템을 만든다"가 아니라 "이미 만들어진 플랫폼에 올린다"가 핵심.

## 발동 조건

- "취미로 돈 벌고 싶어", "온라인 클래스 수익화", "강의/원데이클래스 팔고 싶어"
- "코드 없이 결제 받고 싶어", "구독 모델 만들고 싶은데 개발은 못 해"
- "전자책/PDF/템플릿 팔기", "멤버십·후원 받기"
- `/nocode-monetization`

## 이 skill 을 쓰면 안 되는 경우 (즉시 다른 skill 로)

| 상황 | 보낼 곳 |
|---|---|
| "어떤 수익 모델/가격 구조가 맞을까" 전략 자체가 필요 | `monetization-planner` (전략 → 그 다음 여기로 복귀) |
| 자체 웹/앱에 결제를 **코드로** 붙여야 함 (웹훅·영수증검증·상태머신) | `payment-integrator` |
| 광고로 수익화 | `ad-monetization` |
| 앱스토어 출시·인앱결제 | `store-launcher` |

이 skill 은 **백엔드 불필요**가 전제다. 코드가 필요해지는 순간 곧바로 승격 안내한다(아래 "코드 승격 신호" 참조).

## Workflow

### 1. 진단 (AskUserQuestion)

한 번에 하나씩 묻는다. 답이 명확하면 다음으로.

- **콘텐츠**: 무엇을 파는가? (지식·강의 / 작품·굿즈 / 디지털 파일 / 서비스·시간 / 팬 후원)
- **반복성**: 한 번 사면 끝인가, 계속 받을 게 있는가?
- **대면 여부**: 실시간(줌·오프라인) / 녹화 / 자동 전달?
- **타겟**: 한국 위주 / 해외 포함?
- **규모 감각**: 월 몇 명을 예상하나? (수수료·정산 판단에 필요)
- **기술 한계 확인**: 직접 사이트를 만들 생각은 없는 게 맞나? (있다면 → `payment-integrator`)

### 2. 수익 모델 선택 Decision Tree

```
무엇을 파는가?
├─ 지식·노하우 가르치기
│   ├─ 녹화해서 반복 판매 → 온라인 클래스 (VOD)
│   └─ 실시간으로 직접 → 원데이클래스 / 1:1 레슨
├─ 디지털 파일 (전자책·템플릿·프리셋·악보)
│   └─ 단건 판매 (Digital Download)
├─ 작품·핸드메이드·굿즈
│   └─ 스마트스토어 / 마켓 단건 판매
├─ 서비스·시간 (컨설팅·원고·디자인)
│   └─ 의뢰 단건 (크몽 형 마켓)
└─ 계속 받을 가치가 있다 (정기 콘텐츠·커뮤니티·팬)
    ├─ 매달 새 콘텐츠 → 구독 (Membership)
    └─ 응원·후원 성격 → 후원/멤버십 (부스트·Patreon)
```

선택 결과를 한 줄로 확정: "→ {모델}로 갑니다."

### 3. 운영 플랫폼 매칭

모델이 정해지면 **노코드 운영 플랫폼**을 붙인다. 사용자는 회원가입·콘텐츠 업로드만 하면 결제·정산이 플랫폼에서 끝난다.

| 플랫폼 | 적합 콘텐츠 | 수수료(러프) | 정산 주기 | 한국/해외 | 코딩 |
|---|---|---|---|---|---|
| 클래스101 | VOD 클래스·취미 | 약 30~50% (입점형) | 월 정산 | 한국 | 0 |
| 탈잉 | 실시간/원데이·튜터링 | 약 20~30% | 월~격주 | 한국 | 0 |
| 크몽 | 서비스·재능·의뢰 | 약 10~20% | 정산 신청 후 | 한국 | 0 |
| 네이버 스마트스토어 | 실물·굿즈·디지털 | 약 2~6% + 결제수수료 | 구매확정+1~2일 | 한국 | 0 |
| 부스트(Boosty)/후원 | 팬 후원·멤버십 | 약 5~10% | 월 | 한국/글로벌 | 0 |
| Stripe Payment Links | 단건·구독 (자체 링크) | 약 2.9% + 30¢ | D+7 내외 | 글로벌 | 0 (링크만) |
| Gumroad | 디지털 파일·구독 | 약 10% + 결제수수료 | 주 단위 | 글로벌 | 0 |

> 수수료·정산 주기는 정책이 자주 바뀐다. 추천 직전 **WebSearch/WebFetch 로 해당 플랫폼 최신 수수료·정산 페이지를 확인**하고 숫자를 갱신해 제시한다. 위 표는 출발점이지 확정값이 아니다.

매칭 원칙:
- **노출(트래픽)이 약한 초보** → 입점형(클래스101·탈잉·크몽)로 시작. 수수료 높지만 손님을 데려다준다.
- **이미 팬·채널이 있음** → 자체 링크형(Stripe Payment Links·Gumroad·스마트스토어). 수수료 낮고 고객 데이터를 가져간다.
- 한국 일반 고객 결제 편의 → 간편결제(카카오·네이버) 지원 플랫폼 우선.

### 4. 가격·무료체험·번들 결정

- **앵커 가격**: 같은 카테고리 상위 상품 3개를 WebSearch 로 찾아 가격대 제시. 그 사이에서 포지셔닝.
- **한국 가격 심리**: ₩9,900 / ₩29,000 / ₩49,000 구간. 만원·3만원·5만원 미만 심리선.
- **무료체험/맛보기**: 클래스 1강 무료 공개, 전자책 목차+1장 미리보기 → 전환의 미끼.
- **번들**: 단건 여러 개를 묶어 객단가↑ (예: 강의 + 템플릿 + Q&A 1회).
- **구독이라면**: 월 vs 연(2개월 무료 수준 할인), 무료 7일 체험 여부.

### 5. 이탈·LTV 러프 가정 (계산기만, 코드 없음)

정밀 분석이 아니라 "이 가격이면 먹고 살 만한가"를 가늠하는 봉투 뒷면 계산.

- **단건**: 월 방문 × 구매전환율(보통 1~3%) × 단가 = 월 매출. 수수료·세금 차감 후 실수령 확인.
- **구독**: LTV ≈ 월 구독료 ÷ 월 이탈률(churn). 예) ₩9,900 / 월 이탈 10% → LTV ≈ ₩99,000.
- **체크**: 손님 1명 데려오는 비용(광고·시간)이 LTV보다 작아야 지속 가능.

필요하면 `scripts/ltv_napkin.py` 로 즉석 계산해 숫자를 보여준다(아래 참조).

### 6. 정산·세금 기초 (한국)

- **사업자 등록**: 지속·반복 판매면 사실상 필요. 처음엔 **간이과세자**로 시작 가능(연 매출 기준 충족 시).
- **부가세**: 일반과세 10%. 간이과세는 업종별 낮은 부가율. 플랫폼이 세금계산서·현금영수증을 대신 처리하는지 확인.
- **종합소득세**: 5월 신고. 플랫폼 정산내역 = 소득 증빙으로 보관.
- **해외 플랫폼(Stripe·Gumroad)**: 외화 입금·환율, MoR(Merchant of Record) 여부 확인. Gumroad·Paddle 류는 세금을 대신 떼주기도 함.
- **원칙**: 첫 정산 전에 "플랫폼이 떼는 것(수수료)"과 "내가 신고할 것(소득세)"을 분리해 메모. 세무 디테일은 세무사/홈택스 안내로 연결, 이 skill 은 단정하지 않는다.

### 7. 산출물

코드·파일을 만들지 않는다. 채팅으로 정리해 제시:

1. **선택된 모델** + 한 줄 근거
2. **추천 플랫폼 1순위 + 대안 1개** (최신 수수료·정산 확인 결과 포함)
3. **가격표** (단건/번들/구독 옵션 + 무료 미끼)
4. **러프 매출·LTV 추정**
5. **시작 체크리스트** (가입 → 콘텐츠 업로드 → 가격 등록 → 정산계좌 → 첫 홍보)
6. **세금 메모** (사업자/부가세/소득세 한 줄씩)

## 코드 승격 신호 (→ payment-integrator)

아래가 하나라도 나오면 노코드 한계. `payment-integrator` 로 승격 안내한다.

- 자체 웹사이트/앱에 결제를 **임베드**하고 싶다 (플랫폼 링크로 부족)
- 웹훅으로 결제 성공/실패를 **내 DB·기능에 연동**해야 한다
- 구독 상태에 따라 **앱 기능 잠금/해제**가 필요하다
- 영수증 검증·멱등성·dunning 같은 **결제 엔지니어링**이 필요하다

승격 멘트 예: "여기부터는 플랫폼만으로 안 되고 코드가 필요합니다. `payment-integrator` 로 넘기면 Stripe/PortOne 웹훅과 구독 상태머신을 붙일 수 있어요."

## 검증 체크리스트

- [ ] 사용자가 **코드를 쓰지 않아도** 끝까지 갈 수 있는 경로인가
- [ ] 플랫폼 수수료·정산 주기를 **최신으로 확인**했는가 (표 그대로 복붙 금지)
- [ ] 추천 플랫폼이 사용자의 트래픽 상황(초보 vs 팬보유)과 맞는가
- [ ] 가격이 카테고리 앵커 대비 합리적인가
- [ ] 실수령(수수료·세금 차감 후) 기준으로 매출을 봤는가
- [ ] 한국 결제 편의(간편결제) 고려했는가
- [ ] 코드가 필요한 순간 `payment-integrator` 로 보냈는가

## 완료 보고 (HTML) — 표준
작업을 끝내면 **HTML 완료 보고서**를 생성한다 (SimonKCore `completion-report` 표준).
- 첫 화면은 **심플 요약**(한눈 카드 한 줄) + 직관 그래픽/차트(인라인 SVG)·이미지.
- 각 항목 옆 **[자세히] 버튼**(`<details>`)을 펼치면 상세 — 처음부터 쏟지 않는다(progressive disclosure).
- 자체완결 1파일(인라인 CSS/SVG, 무JS) · 사용자 언어 · 현지시간 스탬프.
- Core 있으면 `completion-report` 호출, 없으면 동일 형식으로 인라인 생성.
